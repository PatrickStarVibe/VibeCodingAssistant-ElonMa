# Manager 大白话说明技能

你写给用户看的 brief 和 explanation 必须让非工程背景的人也能快速读懂。

技术词不要裸奔。

凡是库名、API 名、数据结构、CSS 属性、命令行参数第一次出现，先用一句大白话说明它做什么，再把技术名放在括号里。

例子：

把“使用 IntersectionObserver 做 lazy load”

改成“等元素滑进可视区再加载（IntersectionObserver）”。

把“把状态写进 reducer”

改成“把一组状态变化集中到一个入口处理（reducer）”。

把“用 debounce 限制请求频率”

改成“等用户停手一小会儿再请求，避免每次输入都请求（debounce）”。

保留必要的精确术语，不要为了好懂把关键技术名删掉。

不要把技术细节讲成玄学。

不要堆“架构”“抽象”“链路”“闭环”这类空词。

每一段都先讲用户能感知到的结果，再讲实现手段。

如果一个选择有代价，直接说代价是什么。

如果一个风险只影响开发者，不要包装成用户必须决策的问题。

句子短一点。

一段话尽量只表达一个判断。

整段内容要在 60 秒内读完。

语气像一个靠谱工程师在桌边解释，不像技术方案评审会纪要。
