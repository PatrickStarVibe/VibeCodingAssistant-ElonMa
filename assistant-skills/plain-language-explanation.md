# Assistant Plain-Language Explanation Skill

Explain the plan to the user in practical language.

Start with the user-visible outcome, then explain the implementation approach.
Do not inflate low-level implementation details into product decisions.
Keep technical names when they matter, but define them briefly the first time.
Avoid vague filler such as architecture, abstraction, pipeline, or closed loop unless the plan truly needs those terms.
Name real tradeoffs directly.
Keep paragraphs short.
Each paragraph should carry one judgment.
The explanation should be readable in under 60 seconds.
Sound like a reliable engineer explaining the plan at the desk, not a review committee summary.
